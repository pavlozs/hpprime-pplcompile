# hpprime-pplcompile

Compiler and decompiler between readable HP PPL source files (`.ppl`) and the HP Prime binary program format (`.hpprgm`).

HP Prime stores programs as binary `.hpprgm` files. This toolset lets you write and maintain programs as plain-text `.ppl` files under version control, then compile them for transfer to the calculator via HP Connectivity Kit.

HP Connectivity Kit and the HP Prime Virtual Calculator (emulator) can be downloaded from:
**https://updates.moravia-consulting.com/**

## Requirements

- Python 3.10+
- Standard library only (no external dependencies)

## Compiler ŌĆō `ppl2hpprgm.py`

Reads `.ppl` source files and produces a single `.hpprgm` binary. Each source file becomes one program block; the block name is taken from the filename (without extension).

> **Note:** The compiler is a pure format converter. It wraps source text in the HP Prime binary container without parsing or validating HP PPL syntax. Syntax errors will only surface when the program is loaded on the calculator.

### Usage

**Option A ŌĆō project file (recommended)**

```bash
python src/ppl2hpprgm.py --project geometry.json
```

**Option B ŌĆō explicit file list**

```bash
python src/ppl2hpprgm.py -o build/aviation.hpprgm \
    src/nav/NavTas.ppl \
    src/nav/NavWca.ppl \
    src/nav/NavWind.ppl
```

### Arguments

| Argument | Description |
|----------|-------------|
| `--project FILE` | Read output path and source list from a JSON project file |
| `-o FILE` | Output `.hpprgm` path (required without `--project`) |
| `sources...` | One or more `.ppl` source files (required without `--project`) |

`--project` and `-o` are mutually exclusive.

### Output

```
OK    build/aviation.hpprgm  (3 programs, 4200 bytes)
```

## Decompiler ŌĆō `hpprgm2ppl.py`

Reads a binary `.hpprgm` file and writes one `.ppl` file per program block, plus a project definition file (`.json`).

### Usage

```bash
python src/hpprgm2ppl.py geometry.hpprgm output_dir/
```

### Arguments

| Argument | Description |
|----------|-------------|
| `input` | Input `.hpprgm` file |
| `output_dir` | Directory where `.ppl` files and the project file are written (created if absent) |

### Output

```
OK    output_dir/triVA.ppl  (2006 chars)
OK    output_dir/triS.ppl   (1956 chars)
OK    output_dir/geometry.json
```

## Project definition file

A JSON file that describes which source files build a given `.hpprgm`. Paths are relative to the project file itself.

### Format

```json
{
  "output": "geometry.hpprgm",
  "programs": [
    "triVA.ppl",
    "triS.ppl"
  ]
}
```

| Field | Type | Description |
|-------|------|-------------|
| `output` | string | Output `.hpprgm` filename (relative to the project file) |
| `programs` | array of strings | Source `.ppl` files in compilation order (relative to the project file) |

### Typical workflow

```
project/
Ōö£ŌöĆŌöĆ geometry.json   ŌåÉ project definition
Ōö£ŌöĆŌöĆ triVA.ppl       ŌåÉ source files
Ōö£ŌöĆŌöĆ triS.ppl
ŌööŌöĆŌöĆ geometry.hpprgm ŌåÉ generated by compiler, transferred to HP Prime
```

1. Edit `.ppl` source files in any text editor.
2. Run the compiler: `python src/ppl2hpprgm.py --project geometry.json`
3. Transfer `geometry.hpprgm` to the calculator via HP Connectivity Kit.

The decompiler generates the project file automatically when extracting programs from an existing `.hpprgm`:

```bash
python src/hpprgm2ppl.py existing.hpprgm src/
# ŌåÆ src/ProgramA.ppl
# ŌåÆ src/ProgramB.ppl
# ŌåÆ src/existing.json
```

## Notes on `#pragma mode`

`#pragma mode( separator(.,;) integer(h32) )` is an HP PPL directive that sets:
- decimal separator to `.`, argument separator to `,`, list separator to `;`
- integer arithmetic to 32-bit mode

The directive is stored as part of the source text inside the `.hpprgm` binary.

**Compiler behaviour:** if the `.ppl` source file does not contain a `#pragma mode` line, the compiler automatically prepends the standard pragma shown above. If the source already contains `#pragma mode`, the user's version is used as-is.

**Decompiler behaviour:** the source text is written to `.ppl` exactly as stored in the binary, pragma included.

## .hpprgm binary format

Format reverse-engineered and verified against `tests/samples/export/geometry.hpprgm` exported from HP Prime. Full documentation: [FORMAT.md](FORMAT.md).

In brief: 64-byte global header + N├Ś88-byte catalog section + program data blocks, each containing a UTF-16-LE null-terminated source string.

## Tests

Tests are in `tests/test_roundtrip.py` and use the Python standard `unittest` framework.
Run from the **`hpprime-pplcompile` directory**:

```bash
cd hpprime-pplcompile
python -m unittest tests.test_roundtrip -v
```

### Reference data

| File | Description |
|------|-------------|
| `tests/samples/export/geometry.hpprgm` | Real `.hpprgm` exported from HP Prime via HP Connectivity Kit |
| `tests/samples/src/triVA.ppl` | Source copy-pasted from HP Connectivity Kit editor (no `#pragma`) |
| `tests/samples/src/triS.ppl` | Same |
| `tests/samples/src/geometry.json` | Reference project file generated by the decompiler |

### Test classes

**`TestDecompile`** ŌĆö verifies the decompiler against real HP Prime output

| Test | What it checks |
|------|----------------|
| `test_program_count` | `geometry.hpprgm` contains exactly 2 program blocks |
| `test_program_names` | Extracted names are `triVA` and `triS` |
| `test_triVA_matches_reference` | Decompiled `triVA` source matches `src/triVA.ppl` byte-for-byte |
| `test_triS_matches_reference` | Decompiled `triS` source matches `src/triS.ppl` byte-for-byte |

**`TestProjectFile`** ŌĆö verifies project file generation and consumption

| Test | What it checks |
|------|----------------|
| `test_decompile_generates_project_file` | Decompiler creates a `.json` file in the output directory |
| `test_generated_project_matches_reference` | Generated JSON matches `src/geometry.json` exactly |
| `test_project_output_field` | `output` field equals `geometry.hpprgm` |
| `test_project_programs_field` | `programs` list equals `["triVA.ppl", "triS.ppl"]` |
| `test_load_project_returns_correct_output` | `load_project()` resolves output path correctly |
| `test_load_project_returns_correct_sources` | `load_project()` resolves source paths correctly |
| `test_compile_from_project_file` | Compiler produces a valid `.hpprgm` when driven by a project file |

**`TestCompileRoundtrip`** ŌĆö verifies full compile/decompile symmetry

| Test | What it checks |
|------|----------------|
| `test_roundtrip_all_sources` | Compile `src/*.ppl` ŌåÆ decompile ŌåÆ source content matches originals |
| `test_roundtrip_via_project_file` | Decompile ŌåÆ project file ŌåÆ compile ŌåÆ decompile ŌåÆ matches original sources |
| `test_roundtrip_preserves_unicode` | `Ōēż`, `ŌłÜ` and Czech characters survive UTF-16-LE encode/decode |
